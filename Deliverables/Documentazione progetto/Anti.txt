1. INTRODUZIONE
1.1. Purpose of the system
TrackFit si propone come una piattaforma digitale integrata per il monitoraggio e l'analisi delle attività 
sportive, rivolta a utenti amatoriali e semi-professionisti. Il sistema consente la registrazione 
personalizzata delle sessioni di allenamento, la gestione dei progressi attraverso dashboard interattive, la 
prenotazione di allenatori certificati e la gestione completa degli abbonamenti alle palestre partner.
Il portale, accessibile sia tramite interfaccia web che app mobile, gestisce quattro categorie di utenti: 
Atleta, Coach, Gestore della Palestra (Amministratore di struttura) e Host (Super-amministratore di 
sistema). Ogni ruolo dispone di funzionalità dedicate e di un pannello di controllo personalizzato.
1.2. Design objectives
Il sistema dovrà essere progettato in modo da garantire:
• Usabilità
◦ L'interfaccia utente deve essere responsiva, intuitiva e coerente su tutte le piattaforme (web, 
tablet, mobile). Tutti gli input devono essere validati in tempo reale, con feedback 
immediato all'utente.
• Affidabilità
◦ Nessun errore deve rimanere ingestito. Le sessioni di allenamento devono essere salvate 
localmente in caso di assenza di rete e sincronizzate non appena la connessione è ripristinata. 
Il sistema deve garantire un uptime del 99,5%.
• Sicurezza
◦ Tutte le comunicazioni avvengono tramite HTTPS. Le password sono memorizzate con 
algoritmo di hashing robusto (bcrypt). L'accesso alle risorse è regolato da un sistema RBAC 
(Role-Based Access Control). I pagamenti rispettano lo standard PCI-DSS tramite 
integrazione con gateway certificato (es. Stripe).
• Scalabilità
◦ L'architettura cloud deve supportare picchi di carico simultanei (es. fine gara con migliaia di 
atleti che caricano dati contemporaneamente) mediante auto-scaling orizzontale.
• Interoperabilità
◦ Il sistema deve supportare l'importazione di file GPX/TCX e l'integrazione con API di terze 
parti (Strava, Google Fit) tramite protocollo OAuth 2.0, oltre all'esportazione in formati CSV 
e PDF.

1.3. Object model optimization
Rispetto al modello definito in fase di analisi (RAD v1.2), l'Object Model è stato ottimizzato con le 
seguenti modifiche principali:
• La classe Workout acquisisce l'attributo stato (Pianificata, InCorso, CompletataLocalmente, 
InSincronizzazione, Consolidata) per modellare l'intero ciclo di vita di una sessione, incluso il 
recupero dopo crash del dispositivo.
• La classe Abbonamento acquisisce l'attributo qrCode e il metodo generaQRCode(), necessari per 
il caso d'uso UC9 (Accesso Fisico alla Struttura).
• La classe Coupon viene aggiunta al modello per supportare la logica di sconto (UC8), con 
attributi codice, percentualeSconto, validita, rimosso e il metodo applicaSconto().
• La classe Pagamento viene aggiunta con attributi idPagamento, importo, dataPagamento, stato e 
metodi eseguiPagamento() e generaFattura(), necessari per i casi d'uso UC7 e UC8.
• La classe ListaAttesa viene introdotta per modellare la coda degli utenti in attesa di un posto in 
un corso pieno (FR25), con metodi aggiungiUtente() e notificaUtente().
• Workout acquisisce un metodo recuperaDati() per gestire il ripristino delle sessioni dopo 
spegnimento imprevisto del dispositivo (UC4 alternativa).
2. PACKAGES
2.1. Project structure
La struttura del progetto si presenta nel modo seguente:
• Frontend (Next.js / TypeScript)
◦ src/app — routing e pagine Next.js (App Router)
◦ src/components — componenti React riutilizzabili
◦ src/hooks — custom React hooks
◦ src/store — stato globale (es. Zustand)
◦ src/styles — CSS Modules / Tailwind
• Backend (Next.js API Routes / Node.js)
◦ src/app/api — endpoint RESTful
◦ src/lib — logica di business e servizi
◦ src/db — accesso al database (MySQL)
Il backend è suddiviso in moduli applicativi secondo una separazione in sottosistemi. I servizi 
principali sono organizzati nei seguenti package:

4. DESIGN PATTERNS
4.1. Object model optimization
All'interno dell'applicazione viene adottato un design pattern ibrido che combina il Facade Pattern con 
l'architettura Ports & Adapters (Hexagonal Architecture).
Ogni modulo espone le proprie funzionalità esclusivamente attraverso interfacce di porta (port/in per i 
casi d'uso in ingresso, port/out per i servizi in uscita). Le implementazioni concrete sono incapsulate 
nelle classi ManagerService, rendendo il sistema indipendente dai dettagli tecnologici (database, API 
esterne, gateway di pagamento).
Per ogni entità di dominio principale sono identificati quattro servizi corrispondenti alle operazioni 
CRUD:
• CreateManagerService — gestisce la creazione dell'entità
• ReadManagerService — gestisce la lettura e il recupero dell'entità
• UpdateManagerService — gestisce l'aggiornamento dell'entità
• DeleteManagerService — gestisce la rimozione dell'entità
Questo approccio garantisce la separazione delle responsabilità (SRP), facilita il testing unitario tramite 
mock delle porte e permette la sostituzione delle implementazioni senza impattare il dominio.
4.2. Pattern dei dati persistenti
Per la persistenza dei dati viene adottato il Repository Pattern, che funge da astrazione tra la logica di 
dominio e il layer di accesso ai dati. Nell'implementazione specifica:
• Lato backend Node.js/TypeScript: viene utilizzato Prisma ORM con MySQL, che implementa il 
pattern Repository attraverso le Prisma Client APIs.
• Per l'app mobile (React Native / PWA): viene utilizzato AsyncStorage per la persistenza locale 
delle sessioni in corso, garantendo il funzionamento offline e il recupero dati in caso di crash 
(UC14).
Ogni Repository è definito come interfaccia nel layer domain e implementato nel layer infrastructure, 
seguendo il principio di inversione delle dipendenze (DIP).
4.3. Pattern per le notifiche
Il sistema di notifiche (push, email, in-app) adotta il pattern Observer. Il NotificationService è il soggetto 
osservato; i canali di notifica (Firebase Cloud Messaging, Servizio Email) sono gli osservatori registrati. 
Questo consente di aggiungere nuovi canali senza modificare la logica di business.
Esempi di eventi che scatenano notifiche:
• Promemoria sessione programmata (10 minuti prima dell'inizio)

• Modifica piano allenamento da parte del coach
• Conferma prenotazione corso o slot allenatore
• Disponibilità di un posto in lista d'attesa
• Cancellazione di un corso da parte del gestore
4.4. Pattern per l’interoperabilità
Per l'integrazione con servizi esterni (Strava, Google Fit, gateway di pagamento Stripe) viene adottato il 
pattern Adapter. Ogni integrazione esterna è incapsulata in un adapter che traduce le chiamate del 
dominio nel formato richiesto dall'API di terze parti, utilizzando il protocollo OAuth 2.0 per 
l'autenticazione.
Questo approccio consente di sostituire un provider esterno (es. cambiare gateway di pagamento) 
modificando esclusivamente il relativo adapter, senza impattare la logica di business.
4.5. Pattern per la gestione dello stato offline
La gestione del ciclo di vita delle sessioni di allenamento è modellata attraverso il pattern State. I 
possibili stati di un Workout (Pianificata, InCorso, Interrotta, InPausa, CompletataLocalmente, 
InAttesaDiRete, InSincronizzazione, Consolidata) definiscono le transizioni valide e le azioni permesse 
in ciascuno stato.
La strategia di sincronizzazione (Wi-Fi vs. rete cellulare, priorità della coda) è implementata tramite il 
pattern Strategy, permettendo di adattare il comportamento in base al tipo di connessione disponibile.

React Native (per coerenza con lo stack) o come PWA (Progressive Web App) 
basata su Next.js.
 NFR-I3 (Stack Backend): Le API di backend devono essere sviluppate in 
TypeScript (utilizzando Next.js API Routes o un framework Node.js dedicato) e 
seguire un'architettura RESTful.
 NFR-I4 (Database): Il database principale deve essere un RDBMS (Relational 
Database Management System) MySQL.
 NPR-I5 (Controllo Versione): Il codice sorgente deve essere gestito su un 
repository Git (es. GitHub).
3.3.6 Interface (Requisiti di Interfaccia)
 NFR-IF1 (API Gateway Pagamenti): Il sistema deve interfacciarsi in modo sicuro 
con un gateway di pagamento esterno (es. Stripe) per elaborare gli acquisti degli 
abbonamenti (FR20).
 NFR-IF2 (Importazione Dati): Il sistema deve supportare l'importazione di file in 
formati standard (GPX, TCX) per la registrazione degli allenamenti esterni (FR27).
 NFR-IF3 (Esportazione Dati): Il sistema deve fornire funzionalità di esportazione 
dei report (FR28) in formati standard (CSV, PDF).
3.3.7 Packaging (Requisiti di Distribuzione)
 NFR-PK1 (Accesso Web): L'applicazione web deve essere accessibile 
pubblicamente tramite un nome di dominio sicuro (HTTPS).
 NFR-PK2 (Distribuzione Mobile): L'applicazione mobile (se nativa) deve essere 
distribuibile tramite i principali store ufficiali (Apple App Store e Google Play 
Store).
 NFR-PK3 (Automazione Deploy): Il processo di rilascio (build, test e deploy) deve 
essere automatizzato tramite un sistema di CI/CD (es. GitHub Actions).
3.3.8 Legal (Requisiti Legali e di Conformità)
 NFR-L1 (GDPR): Il sistema deve essere conforme al Regolamento Generale sulla 
Protezione dei Dati (GDPR) per la raccolta, l'archiviazione e il trattamento dei dati 
personali degli utenti, inclusa la gestione del consenso e il diritto alla cancellazione 
(FR30).
 NFR-L2 (Conformità Pagamenti): Il sistema deve aderire agli standard di 
sicurezza PCI-DSS. L'integrazione con il gateway di pagamento (NFR-IF1) deve 
essere implementata in modo da ridurre al minimo l'ambito di conformità (es. 
utilizzando componenti UI ospitati dal provider).
 NFR-L3 (Termini e Condizioni): Gli utenti devono poter visualizzare e accettare 
esplicitamente i Termini di Servizio e l'Informativa sulla Privacy al momento della 
registrazione.

3.3.9 Vincoli di dominio
 R1: Una sessione dell’atleta è modificabile solo se l’ora di inizio dista ≥ 48 ore dal 
momento della modifica; altrimenti l’operazione è bloccata.
 R2: Ogni modifica approvata invia automaticamente un’e-mail all’atleta con 
riepilogo, motivazione e riferimento al piano.
 R3: Cancellazione gratuita ≥ 24 ore; penale 50% oltre il limite.
 R4: Coupon valido solo per la struttura “FitCorp — Roma Centro” e piano 
“Annuale”; un solo utilizzo per utente; scadenza 31/10/2025.
 R5: Sconto applicato solo alla quota abbonamento (non cumulabile).
 R6: Pagamento con Strong Customer Authentication (3-D Secure).
 R7: Tessera digitale valida ai tornelli solo con SUB “Attivo”.
 R8: Unicità struttura (P.IVA e CUN come chiavi globali univoche).
 R9: Anti-duplicato fuzzy su Denominazione + Indirizzo con punteggio di similarità.
 R10: Tutte le operazioni amministrative (incluse coupon, pagamento, creazione 
SUB, fattura) registrano un audit (utente, timestamp, azione).

1. Introduzione
In questo documento descriveremo come si intende procedere con l'architettura del 
sistema TrackFit e quali sono gli obiettivi di design che intendiamo raggiungere.
1.1 Scopo del sistema
TrackFit si propone come una piattaforma digitale per il monitoraggio e la gestione 
delle attività sportive, rivolta a utenti amatoriali e semi-professionisti. La piattaforma 
consente la registrazione e l'analisi delle sessioni di allenamento, la gestione delle 
palestre e degli abbonamenti, il monitoraggio delle performance e la generazione di 
reportistica personalizzata.
Il sistema è accessibile tramite interfaccia web (Flutter) e applicazione mobile , e 
gestisce quattro categorie principali di attori:
 Utente/Atleta: registra allenamenti, prenota corsi, gestisce il proprio 
abbonamento.
 Coach/Allenatore: pianifica e modifica i piani di allenamento degli atleti 
associati.
 Gestore della palestra: amministra struttura, corsi, orari e gestisce i report 
gestionali.
 Amministratore: supervisiona tutte le strutture connesse al sistema con 
visione aggregata.
1.2 Design Goals
Gli utenti principali di TrackFit sono gli atleti e i gestori di palestre. È necessario 
fornire un'esperienza d'uso semplice e intuitiva, garantendo al contempo affidabilità 
nella gestione di dati sensibili (parametri fisici, pagamenti) e scalabilità per gestire 
picchi di carico durante eventi sportivi. In base a queste osservazioni, si identificano i
seguenti obiettivi di design:
 Disponibilità: il sistema deve essere disponibile h24, 7 giorni su 7, con un 
uptime garantito del 99,5%. Gli atleti devono poter registrare un allenamento o 
prenotare un corso in qualsiasi momento, indipendentemente dal fuso orario.
 Scalabilità: il sistema deve supportare almeno 1000 utenti concorrenti 
mantenendo i tempi di risposta sotto i 2 secondi (NFR-P1, NFR-P3). 
L'architettura deve permettere la scalabilità orizzontale per gestire picchi di 
carico, ad esempio durante eventi sportivi di massa (scenario maratona, UC5).
 Sicurezza: TrackFit gestisce dati personali sensibili (parametri fisici, 
documenti), dati finanziari (pagamenti, abbonamenti) e credenziali di accesso. 
Il sistema deve essere conforme al GDPR (NFR-L1) e agli standard PCI-DSS 
(NFR-L2). Tutte le comunicazioni devono avvenire tramite HTTPS e le 
password devono essere cifrate con bcrypt (NFR-R3, NFR-R5).
 Gestione dei malfunzionamenti: dovendo gestire sessioni di allenamento in 
corso e transazioni economiche, i malfunzionamenti devono essere gestiti 
garantendo l'integrità dei dati. Particolare attenzione va al rispetto delle 
proprietà ACID nelle transazioni di pagamento e al recupero delle sessioni 
interrotte (scenario crash batteria, UC4 alternativa).
 Performance: le operazioni standard (login, visualizzazione dashboard, 
apertura pagine) devono essere completate in meno di 2 secondi (NFR-P1). La 
generazione di report complessi deve essere completata in meno di 5 secondi 
(NFR-P2). Il sistema deve implementare strategie di caching per le query 
frequenti.

1.4 Definizioni, acronimi e abbreviazioni
Acronimi:
• RAD = Requirement Analysis Document
• SDD = System Design Document
• COTS = Commercial Off-the-Shelf
Definizioni:
• Sottosistema: corrisponde alla parte di lavoro che può essere svolta autonomamente 
da un singolo sviluppatore o da un gruppo di sviluppatori. Si ottiene decomponendo il
sistema.
• Design Goal: proprietà del sistema sulla quale ci si concentra maggiormente.
• Trade-off: possibilità di ridurre una certa qualità per aumentare il valore di un’altra
qualità e viceversa. Il termine è espresso talvolta come costo opportunità, riferendosi 
a più alternative alle quali si è preferito rinunciare a vantaggio di un’altra scelta.
• Mapping Hardware-Software: descritto per indicare i vari device hardware 
utilizzati dal sistema e la loro interazione con le componenti software.
• Dati Persistenti: dati che devono sopravvivere all’esecuzione singola 
dell’applicazione. Sono i dati che vengono salvati nel database.

2. Architettura del sistema corrente
Attualmente non esiste un sistema unificato che consenta simultaneamente:
 monitoraggio degli atleti,
 gestione degli allenatori,
 analisi statistiche della palestra,
 comunicazione tra utenti sportivi.
Le soluzioni esistenti gestiscono solo singoli aspetti (fitness tracking personale 
oppure gestione amministrativa delle palestre), senza integrare tutte le funzionalità in 
un’unica piattaforma.
TrackFit nasce quindi come sistema integrato per la gestione completa 
dell’ecosistema sportivo.
3. Architettura del sistema proposto
3.1 Panoramica
Nelle prossime sezioni andremo a definire l'architettura del sistema in termini di:
 Decomposizione in sottosistemi: si propone un'architettura a tre livelli con 
suddivisione tra interfacce utente, logica applicativa e logica di persistenza.
 Hardware/software mapping: distribuzione dei componenti software sui 
dispositivi fisici (client, server cloud, database).
 Gestione dei dati persistenti: struttura e tecnologie utilizzate per il 
salvataggio dei dati.
 Controllo di accessi e sicurezza: tabella degli accessi per ruolo e meccanismi 
di autenticazione/autorizzazione.
 Controllo globale del software: approccio al control flow e gestione della 
concorrenza.
 Boundary conditions: gestione dei casi limite e degli scenari di errore.
3.2 Decomposizione in Sottosistemi
Per la divisione in sottosistemi abbiamo analizzato i requisiti funzionali e non 
funzionali del RAD. Considerando la natura multi-ruolo e multi-piattaforma (web + 
mobile) di TrackFit, abbiamo optato per un'architettura a tre livelli:
1. Le interfacce degli utenti (divise per ruolo e per piattaforma: Web e Mobile)
2. La logica applicativa (sottosistemi per dominio funzionale)
3. La logica di persistenza (Storage)
Le interfacce utente comunicano tramite API RESTful con i sottosistemi del livello 
applicativo. I permessi di accesso sono descritti nella tabella degli accessi (sez. 3.5).
I sottosistemi della logica applicativa individuati sono:
 UserManagement: gestisce registrazione, autenticazione, profili e ruoli (FR1–
FR4, FR29–FR30).
 GymManagement: gestisce palestre, corsi, orari e capacità (FR5–FR8).
 WorkoutTracking: gestisce la pianificazione, l'esecuzione e il consolidamento
delle sessioni di allenamento (FR9–FR11, FR23–FR25).
 ReportingManagement: gestisce la generazione di report e dashboard per tutti
i ruoli (FR12–FR15, FR28).
 PaymentManagement: gestisce abbonamenti, pagamenti, fatturazione e 
coupon (FR7, FR20–FR22).
 InteroperabilityManagement: gestisce l'importazione/esportazione dati in 
formati standard (FR27–FR28).
I sottosistemi della logica applicativa comunicano con il componente Storage per il 
salvataggio persistente dei dati. Componenti allo stesso livello possono comunicare 
tra loro dove necessario (es. WorkoutTracking con ReportingManagement).

3.3 Mapping Hardware / Software
TrackFit nasce come sistema distribuito cloud-native, in cui client web e mobile da 
qualsiasi dispositivo si connettono a server centralizzati che gestiscono le richieste 
tramite API RESTful. È compito del server verificare i permessi dell'utente 
autenticato prima di eseguire qualsiasi operazione (NFR-R4).
Il sistema sarà sviluppato utilizzando il framework Flutter , che facilita la creazione 
di applicazioni web server-side rendering (SSR) e static generation, riducendo al 
minimo la necessità di configurazioni manuali. Flutter è ideale per implementare sia 
il frontend che le API Routes del backend, offrendo una gestione efficiente del 
routing e delle dipendenze. Per garantire un controllo sicuro degli accessi e la 
gestione dei token di sessione, verrà utilizzato JWT (JSON Web Token) integrato con 
middleware Next.js. Sul lato mobile verrà impiegato React Native, che permette di 
sviluppare applicazioni native iOS e Android condividendo la logica applicativa con 
il frontend web.
La distribuzione avverrà su un provider Cloud (es. AWS o Vercel) con le seguenti 
componenti hardware/software:

3.4 Gestione Dati Persistenti
Il sistema dovrà gestire il salvataggio persistente di:
 I dati degli utenti (profilo, credenziali cifrate, parametri fisici)
 Le sessioni di allenamento (metriche, tracciati GPS, note, tag)
 I piani di allenamento e i calendari degli atleti
 Le strutture, i corsi e gli orari delle palestre
 Gli abbonamenti e lo storico dei pagamenti
 Le fatture, i coupon e i QR code per l'accesso
 I report generati e le impostazioni degli obiettivi
 I log di audit per tutte le operazioni amministrative (R10)
 Le liste d'attesa per i corsi al completo
Per farlo verrà utilizzato un database relazionale MySQL, in modo da garantire 
accesso veloce ai dati e il rispetto delle proprietà ACID nelle transazioni, in 
particolare per le operazioni di pagamento e aggiornamento degli abbonamenti (NFR R2). La comunicazione con il database avverrà tramite un ORM (es. Prisma o 
TypeORM) per garantire type-safety e migrazioni controllate.
Per migliorare le performance (NFR-P1, NFR-P2) verrà integrato un layer di caching 
con Redis per le query frequenti (dashboard, elenco corsi, quote disponibili) e per la

Verranno eseguiti backup automatici giornalieri del database (NFR-R2), con RPO

3.6 Controllo del Flusso Globale
In TrackFit, il control flow sarà implementato utilizzando un approccio event-driven 
e request-response, supportato dall'architettura nativa di Flutter. Le richieste generate 
dai client (HTTP/HTTPS) vengono ricevute dal router di Flutter, che le instrada al 
corretto API Route Handler. All'interno degli handler, i servizi applicativi necessari 
vengono invocati per elaborare le azioni richieste, centralizzando l'elaborazione degli 
input e mantenendo una struttura organizzata a livelli.
3.7 Condizioni Boundary
Le condizioni di confine in TrackFit si riferiscono a tutti i casi limite o alle situazioni 
estreme che potrebbero influenzare il comportamento del sistema. Il monitoraggio e 
la gestione di queste condizioni sono affidati a un amministratore di sistema che 
supervisiona l'infrastruttura cloud, esegue operazioni di manutenzione e interviene in 
caso di malfunzionamenti.
Le principali condizioni di confine identificate sono:
 Crash del dispositivo mobile durante il tracciamento: la sessione viene 
salvata localmente ogni 30 secondi su file temporaneo. Al riavvio dell'app, il 
sistema rileva l'anomalia e propone il recupero dei dati (UC4 alternativa, 
scenario trekking).
 Assenza di rete durante la sincronizzazione: il sistema sospende l'upload, 
imposta lo stato della sessione in 'In attesa di sincronizzazione' e avvia un 
servizio in background che monitora il ripristino della connessione (UC14).
 Picchi di carico simultaneo: l'architettura cloud con auto-scaling e code di 
messaggi (Redis/Bull) garantisce che migliaia di sessioni simultanee vengano 
accodate e processate senza perdite di dati (NFR-P3, NFR-P4).
 Transazione di pagamento fallita: il sistema annulla immediatamente la 
risorsa creata (prenotazione o abbonamento), sblocca lo slot e notifica l'utente 
senza addebiti (UC16).
 Tentativo di creazione struttura duplicata: il sistema blocca la scrittura sul 
DB prima del commit, segnala il campo in conflitto all'amministratore e 
preserva l'integrità del database (UC13, R8, R9).
 Violazione del vincolo di preavviso 48h: il sistema blocca l'operazione del 
Coach prima della scrittura e offre l'alternativa di aggiungere una nota all'atleta
(UC15, R1).
 Deployment e manutenzione: il sistema supporta zero-downtime deploy 
tramite strategie blue-green o rolling update gestite dal provider cloud (NFR P5).
